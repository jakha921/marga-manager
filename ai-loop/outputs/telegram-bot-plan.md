# Telegram Bot Brainstorm

## Recommended Direction

Use `aiogram` v3 as a separate async worker, but do not add it in the current diff. The smallest production path is polling in one dedicated Coolify process first; webhook can come later after the bot URL, TLS, and secret-token flow are stable.

## First Useful Functions

- `/start` — starts binding flow and explains available commands.
- `/link <code>` — binds Telegram chat to an existing Marga user/org with a one-time code generated inside the app.
- `/report` — today's organization summary: entries count, incoming total, stock entries, sales, top kitchens.
- `/report yesterday` — same summary for yesterday.
- `/kitchens` — short list of kitchens and today's activity count.
- `/operations` — last 10 operations for the user's organization.
- `/subscription` — current plan, expiry date, grace-period warning.
- `/daily_on` and `/daily_off` — opt in/out of daily report.
- `/help` — command list.

Avoid free-form database questions in v1. Keep commands deterministic and mapped to existing service/query code.

## Architecture

- Django remains the source of truth.
- Add a small app later, e.g. `apps/telegram_reports`.
- Add model later: `TelegramChatBinding(user, organization, chat_id, language, daily_enabled, verified_at)`.
- Add management command later: `uv run python manage.py run_telegram_bot`.
- Add Celery task later: `send_daily_telegram_reports_task`, scheduled per tenant/admin preference.
- Bot handlers should call shared Django report services, not raw SQL or duplicated analytics logic.
- Daily reports should be sent only to verified tenant admins by default. Kitchen users can be enabled later with restricted scope.

## Security Rules

- Bot token only from env, never in code/logs.
- One-time link code: short TTL, single use, generated by authenticated web user.
- No reports to unbound chats.
- Group chats disabled by default unless explicitly approved by tenant admin.
- Rate limit commands per chat/user.
- Audit binding/unbinding and report delivery failures.
- Do not expose Payme transaction internals in Telegram; show subscription status only.

## Implementation Phases

1. Spec and service layer: define report payloads and reuse existing analytics queries.
2. Binding flow: app-generated one-time code, bot stores chat binding.
3. On-demand commands: `/report`, `/operations`, `/subscription`.
4. Daily Celery report: opt-in schedule, retry/logging.
5. Optional webhook mode: only after deployment environment is stable.

## Open Decisions

- Which roles should receive daily reports by default: TENANT_ADMIN only, or also selected KITCHEN_USER accounts.
- Report time and timezone per tenant.
- Whether clients want Telegram group reports or only private chats.
- Exact language defaults: use user/app language or separate Telegram preference.
